# dms-workspace - Epic Breakdown (2026-03-21)

## Overview

This document provides epic and story breakdown for bug fixes and improvements discovered after Epic 8 (Tailwind CSS Migration). These issues emerged from the Tailwind CSS refactor and require remediation.

## Requirements Inventory

### Functional Requirements

**FR9:** The Storybook stories must render without errors and display component functionality correctly without "Cannot destructure property 'id' of 'defaultExport' as it is undefined" errors.

**FR10:** Loading/wait spinners must be centered both horizontally and vertically on the screen when displayed during data refresh operations.

**FR11:** File upload controls must have clear button-like styling with background color, padding, border radius, and hover effects to indicate they are interactive elements.

**FR12:** Account screens must display all data tables (Open Positions, Sold Positions, Dividend Deposits) correctly and visibly on the screen.

**FR13:** Global Summary screen must display the line graph to the right of summary information on desktop screens, with responsive stacking on smaller screens.

**FR14:** Account Summary screen must display the line graph to the right of summary information on desktop screens, with responsive stacking on smaller screens.

**FR15:** Universe table Position column must display numeric values with 2 decimal places, comma thousands separator, and right alignment, with sufficient column width to show values up to 100,000.00 without clipping.

### Non-Functional Requirements

_No new NFRs - these are bug fixes addressing visual regression and UX issues from Epic 8._

### Additional Requirements

- **Context from Epic 8:** Most issues stem from the Tailwind CSS migration completed in Epic 8. Previous styling using CSS classes and Angular Material directives was refactored to Tailwind utilities, which inadvertently broke these UI elements.
- **Verification requirement:** All fixes must be verified using the Playwright MCP server to ensure proper rendering and behavior.
- **Historical reference:** Git commit history from Epic 8 should be consulted to understand what styles were being used before the Tailwind migration.

### UX Design Requirements

_No formal UX design document exists for these bug fixes - requirements are based on restoring previous functionality._

### Story 15.2: [Implementation] Apply Formatting and Sizing to Position Column

As a user,
I want the Position column to display numbers clearly formatted and fully visible,
So that I can quickly read position sizes without having to resize columns or guess at clipped values.

**Acceptance Criteria:**

**Given** I am viewing the Universe table
**When** I look at the Position column
**Then** all numeric values display with exactly 2 decimal places
**And** values of 1,000 or greater display comma separators (e.g., "100,000.00")
**And** all values are right-aligned within the column

**Given** a position with value 100,000.00
**When** it is displayed in the Position column
**Then** the entire value is visible without clipping
**And** there is sufficient padding on the right side
**And** the column width is wide enough to accommodate the value

**Given** the Angular component template or TypeScript format logic
**When** I implement the number formatting
**Then** I use Angular's DecimalPipe or a custom formatter with pattern '1.2-2'
**And** the formatter includes thousands separator logic
**And** the column CSS includes `text-align: right` or Tailwind's `text-right` class

**When** I set the column width
**Then** the width accommodates at least 10 characters plus padding
**And** the width can be specified in pixels, rem, or a flex-basis value
**And** the column does not shrink below the minimum width on narrow screens

**Then** values under 1,000 display like "500.00" (no comma)
**And** values of 1,000 or more display like "1,234.56" or "100,000.00"
**And** no values are clipped or truncated
**And** all values are right-aligned
**And** the column width is visually appropriate relative to other columns
