# dms-workspace — Epic Breakdown (2026-05-23)

## Overview

Three new epics derived from the 2026-05-23 epic descriptions file. This batch extends the project backlog from **Epic 109** through **Epic 111**. Two are small server/UI scoping fixes on the Universe screen; the third is a substantial cross-cutting refactor of the base table component to finally eliminate sticky-header scroll jank by separating headers from the virtualized body.

- **Epic 109 — Filter Expired Symbols With No Open Positions from Universe Screen:** Expired symbols clutter the Universe screen when Dave can no longer trade them. The server should drop any expired symbol that has no currently open position before sending results to the client. This is a server-side filter applied as Universe rows are assembled.

- **Epic 110 — Restrict Universe Delete Button to Truly Unused Symbols Under "All Accounts" Filter Only:** Today the delete button on a Universe row shows whenever the symbol has no open positions under the *current* account filter. That is too aggressive — a symbol used in trades or dividend deposits for another account would be wrongly deletable. The button must only appear when (a) the active account filter is **All Accounts**, and (b) the symbol is referenced in neither the `trades` table nor the `divDeposits` table across all accounts. Under any specific-account filter, no delete button is shown.

- **Epic 111 — Eliminate Janky Scroll by Decoupling Sticky Header from Virtualized Body (Round 10):** Scrolling jank has been chased through Epics 29, 31, 44, 60, 64, 87, 101, 105, and 106 with sticky-header-based approaches. Investigation shows that **removing sticky headers eliminates the problem entirely**, and the wider community has reached the same conclusion: the only durable fix is to separate the header from the scrollable body, give both fixed-width columns, and synchronize horizontal scrolling between them when the body is wider than the viewport. This epic rebuilds the base table component used across every list screen in the app to use this two-region layout, then verifies every screen behaves correctly, with regression coverage.

All stories follow existing project conventions:

- **Angular 21 zoneless** — `inject()`, `OnPush`, signal-first
- **SmartNgRX / SmartSignals** for state management
- **Fastify** backend, **Prisma** ORM (SQLite dev / PostgreSQL prod)
- **Vitest** for unit tests, **Playwright** for E2E (Chromium + Firefox)
- **Tests are authoritative** — do not change tests unless a test is asserting something we are explicitly changing
- **`pnpm all`** must pass after every story

### Additional Requirements (Architecture / Context)

- **Server-side filtering for Universe (Epic 109):** The Universe endpoint assembles its row list on the server. The filter for "expired AND no open positions" should be applied at the data layer (Prisma query or post-query filter in the Fastify route) — not on the client — so the client never receives the rows. "Open position" means a `trades` row where the position is not yet closed (sell date unset or sell price zero, per existing open/closed convention). "Expired" is determined by the symbol's expiration date being in the past relative to the server's current date.

- **Delete-button gating (Epic 110):** The Universe screen already knows its current account filter (active account vs **All Accounts**). The "symbol is not referenced anywhere" check requires either (a) a server-side derived flag returned per Universe row (`deletable: boolean`), or (b) two `COUNT` queries against `trades` and `divDeposits` keyed by symbol. Option (a) is preferred because it keeps client logic trivial and avoids an N+1 lookup. Story 110.1 should choose and document the approach.

- **Base table refactor (Epic 111):** Prior scrolling epics (29, 31, 44, 60, 64, 87, 101, 105, 106) all attempted to keep `position: sticky` for the header and patch the surrounding CSS/virtual-scroll behaviour. Dave's investigation confirmed that simply removing sticky headers eliminates the artifacts entirely, and the GitHub issue trail for similar virtual-scroll + sticky-header setups converges on the same conclusion: **don't use sticky headers**. The replacement design has two stacked DOM regions:
  1. **Header region** — a single-row table-like container with explicit fixed column widths, placed above the body.
  2. **Body region** — the existing `cdk-virtual-scroll-viewport` (vertical scrolling only) with the same explicit fixed column widths.
  Both regions live inside a horizontally-scrolling outer container when total column width exceeds viewport width. Horizontal scroll is shared between header and body via either (a) wrapping both inside a single `overflow-x: auto` container, or (b) explicitly synchronizing `scrollLeft` between two separate scrollers. Approach (a) is simpler and preferred if it does not break the virtual scroller's vertical scrolling.

  Column widths must be defined once (in the column model) and applied to both header cells and body cells. This eliminates today's auto-fit width drift between header and body and removes any need for `position: sticky`.

  Because the base table is used on every list screen in the app, Story 111.3 explicitly verifies each consuming screen end-to-end before the refactor is considered done. Story 111.4 adds a regression suite that fails CI if sticky headers reappear or if header/body horizontal scroll positions diverge.
