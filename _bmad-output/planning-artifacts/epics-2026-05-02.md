# dms-workspace ŌĆö Epic Breakdown (2026-05-02)

## Overview

This planning artifact references the same five epics (92ŌĆō96) already captured in
`epics-2026-04-30.md`. The `epics-2026-05-02.md` planning file describes the same five topics
with the same scope; no new epics are introduced. All stories (92-1 through 96-2) already exist
as implementation artifact `.md` files with `Status: Approved` and are ready for development.

- **Epic 92 ŌĆö Grab Volatility During Import:** The CUSIP-resolution path in the CSV import
  pipeline (`create-universe-entry.helper.ts`) calls `fetchAndUpdatePriceData` but never calls
  `recalculateUniverseVolatility`. As a result, any symbol added via the Fidelity CSV import
  has `volatility_long`, `volatility_short`, and `volatility_calculated_at` permanently unset
  until the next manual per-symbol edit or batch field update (if Epic 93 is completed). The
  fix mirrors the pattern already used in `add-symbol.function.ts`: call `getDistributions`,
  pass the full history to `recalculateUniverseVolatility`, and supply the prefetched outcome
  to `fetchAndUpdatePriceData`.

- **Epic 93 ŌĆö Grab Volatility During Update Fields:** The batch field-update handler
  (`settings/update/index.ts ŌåÆ processUniverse`) calls `getDistributions` to refresh price
  and ex-date data but completely discards the `history` array and never calls
  `recalculateUniverseVolatility`. Clicking "Update Fields" therefore silently leaves all
  volatility figures stale. The fix is to extract `history` from the `getDistributions`
  outcome and call `recalculateUniverseVolatility(universe.id, history)` for each symbol
  processed in the batch.

- **Epic 94 ŌĆö Return Symbol with divDeposit Data:** The `/api/div-deposits` POST endpoint
  returns `universeId` in every `DivDeposit` row. The client builds a `buildUniverseMap()`
  computed signal that iterates the entire universe store and constructs a
  `Map<universeId, Universe>`, which the dividend-deposits component service then uses to
  resolve each deposit's ticker symbol. Returning the resolved `symbol` string directly from
  the server eliminates the client-side map lookup, simplifies the state model, and removes
  the dependency on the universe store being fully loaded before deposits can display.

- **Epic 95 ŌĆö Return Symbol with Open Positions Data:** The `Trade` interface on both client
  and server already has a `symbol?: string` field, and the SmartNgRX POST endpoint at
  `/api/trades/` returns `symbol` by joining with the `universe` table. However,
  `open-positions-component.service.ts` still calls `buildUniverseMap()` and performs a
  `universeMap.get(trade.universeId)` lookup rather than reading `trade.symbol` directly.
  This epic enforces the symbol field as always-present and removes the universe-map
  dependency from the open-positions component service.

- **Epic 96 ŌĆö Return Symbol with Sold Positions Data:** The same situation as Epic 95 applies
  to the sold-positions component service: `sold-positions-component.service.ts` also uses
  `buildUniverseMap().get(trade.universeId)` instead of `trade.symbol`. After this epic and
  Epics 94ŌĆō95, `buildUniverseMap` has no remaining callers and is deleted entirely.

All stories follow existing project conventions:

- **Angular 21 zoneless** ŌĆö `inject()`, `OnPush`, signal-first
- **SmartNgRX / SmartSignals** for state management
- **Fastify** backend, **Prisma** ORM (SQLite dev / PostgreSQL prod)
- **Vitest** for unit tests, **Playwright** for E2E
- **Tests are authoritative** ŌĆö do not change tests unless a test is asserting something we
  are explicitly changing
- **`pnpm all`** must pass after every story
- **Playwright MCP server** must be used to reproduce and verify all UI failures
- **TDD** ŌĆö failing test first, then implementation, then regression-prevention coverage
