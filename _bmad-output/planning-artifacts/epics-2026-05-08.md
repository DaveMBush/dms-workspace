# dms-workspace — Epic Breakdown (2026-05-08)

## Overview

Four new epics derived from the 2026-05-08 epic descriptions file. This batch extends the project
backlog starting at **Epic 99** through **Epic 102**. The work covers:

- **Epic 99 — Last $ Column on Open Positions Tab:** The `Last $` column on the Open Positions
  tab is empty. The price exists in the `Universe` table and the server already joins symbol
  → universe data for Open Positions (per Epic 95). Either the field was never wired into
  `mapTradeToResponse` (the Epic 97 follow-on that should have caught it), or the client
  stopped reading it after the symbol-on-server refactor. This epic finishes the wiring and
  proves it with an E2E test.

- **Epic 100 — Delete Row on Universe Screen:** Clicking the trash-can icon on a universe row
  reports success in the UI but the row remains visible — and is still present after a hard
  refresh. The server delete is silently failing (or the response is misleading the client).
  This epic reproduces the bug, fixes the delete pipeline end-to-end, and adds regression
  coverage.

- **Epic 101 — General Scrolling Issues (Round 7):** Across all screens, headers occasionally
  scroll under their parent header during slow scrolling, content flickers, and headers can
  end up scrolled below the content. The Angular Material CDK virtual-scroll row height has
  been verified, so the cause is something else. Prior attempts (Epics 29, 31, 44, 60, 64, 87)
  reduced symptoms but did not eliminate them. This epic does a fresh reproduction across every
  scrollable surface, identifies the actual root cause, applies the fix, and adds a regression
  suite that catches the specific failure modes.

- **Epic 102 — Electron App Build Bugs:** The packaged Electron app is currently unrunnable.
  Two distinct defects must be fixed before release: (1) the installer must set
  `/opt/DMS/chrome-sandbox` to `root:root` mode `4755` so Chromium's SUID sandbox helper does
  not abort on launch, and (2) the bundled main-process code crashes on `Cannot find module
  'tslib'`, fixed by setting `"importHelpers": false` in the Electron app's `tsconfig.json` so
  TypeScript inlines the helpers instead of importing them.

All stories follow existing project conventions:

- **Angular 21 zoneless** — `inject()`, `OnPush`, signal-first
- **SmartNgRX / SmartSignals** for state management
- **Fastify** backend, **Prisma** ORM (SQLite dev / PostgreSQL prod)
- **Vitest** for unit tests, **Playwright** for E2E (Chromium + Firefox)
- **Tests are authoritative** — do not change tests unless a test is asserting something we
  are explicitly changing
- **`pnpm all`** must pass after every story
- **Playwright MCP server** must be used to reproduce and verify all UI failures

### Additional Requirements (Architecture / Context)

- **Open Positions data flow (post Epic 95/97):** Trades are joined to `Universe` server-side
  in `mapTradeToResponse` (`apps/server/src/app/routes/trades/index.ts`). Any new computed or
  passthrough field must be added there, exposed in the `Trade` DTO consumed by the client,
  and read from the joined row — not from a client-side `universe.map` lookup (Epic 96 deleted
  `buildUniverseMap`). Verify whether `Last $` was missed in the Epic 97 sweep that restored
  Expected$, Last$ Unrlz Gain%, Unrlz Gain$, Target Gain, and Target Sell to the server.

- **Universe delete pipeline:** The trash-can action on the universe screen calls a server
  delete endpoint. Audit (a) whether the endpoint actually deletes the row, (b) whether the
  HTTP response code is honest, (c) whether SmartNgRX/SmartSignals removes the row from the
  store on success, and (d) whether subsequent universe queries exclude the deleted symbol.
  The bug could be at any of those layers.

- **Scrolling root-cause search space (after 6 prior attempts):** Likely culprits beyond
  CDK row height: subpixel rounding from Tailwind utilities on the header row; `position:
  sticky` losing its containing block when an ancestor has `transform`/`will-change`/
  `contain`; CDK `cdk-virtual-scroll-viewport` wrapping the wrong element relative to the
  sticky header; `OnPush` not flushing layout signals during the scroll frame; or the recent
  symbol-on-server refactor changing row-data identity and triggering unnecessary
  `trackBy` re-renders mid-scroll. Investigation must enumerate which of these are actually
  in play before coding a fix.

- **Electron packaging — chrome-sandbox:** Electron documents the requirement
  (https://www.electronjs.org/docs/latest/tutorial/sandbox#linux-sandbox-helper). The fix is
  delivered through the installer (`electron-builder`'s linux `afterInstall` hook for `.deb`
  and an equivalent for any other formats) so the chmod runs as root at install time. The
  source file inside the unpacked app must be owned by `root:root` with mode `4755`.

- **Electron packaging — tslib:** With `"importHelpers": true`, TS emits `require("tslib")`,
  which the Electron main bundle cannot resolve at runtime because `tslib` was not bundled
  into `app.asar`. The simplest, lowest-risk fix is `"importHelpers": false` in the Electron
  app's `tsconfig.json` so helpers are inlined. (Alternative — adding `tslib` to the
  Electron app's `dependencies` and letting `electron-builder` include it — is intentionally
  not chosen here per the issue reporter's recommendation.)
