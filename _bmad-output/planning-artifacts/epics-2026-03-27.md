# dms-workspace — Epic Breakdown (2026-03-27)

## Overview

This document provides epic and story breakdown for new requirements received on 2026-03-27. These are additive epics that build on prior Epics 1–20. Epic numbering starts at 21.

---

## Requirements Inventory

### Functional Requirements

**FR21:** AI coding agent prompts must include instructions that prevent rate-limit exhaustion when interacting with the GitHub Copilot API. Agents should not call the API more frequently than once every 2 minutes (or as otherwise permitted), and must include wait/delay directives to ensure tasks run to completion even if they run slowly.

**FR22:** All unit tests must achieve 100% branch coverage across the codebase. The `vitest` coverage configuration must enforce a branch-coverage threshold of 100. The `all` script in `package.json` (and any equivalent test-only scripts) must always run coverage so that the quality gate is checked on every test run.

**FR23a:** The `.jscpd.json` file must have the following ignore paths removed (they were added incorrectly and suppress real duplication that should be fixed):
- `**/global/global-screener/**`
- `**/global/global-universe/**`
- `**/global/global-summary.ts`
- `**/global/global-summary.html`
- `**/accounts/account-summary/account-summary.ts`
- `**/accounts/account-summary/account-summary.html`
- `**/routes/trades/get-closed-trades/**`
- `**/routes/trades/get-open-trades/**`

Any duplication that is exposed after removal must be resolved by refactoring the duplicated code.

**FR23b:** The Architecture document (`_bmad-output/planning-artifacts/architecture.md`) must be updated with a Quality Tool Governance section that explicitly prohibits altering the configuration of quality tools (jscpd, eslint, prettier, and any others) without explicit user approval obtained via the `prompt` skill, and requires all approved changes to be documented in the Architecture document.

**FR24:** The Universe screen table must support multi-column sorting of at least 2 simultaneous sort columns. The secondary sort should be applied within results that share the same primary sort value. The UI must indicate which columns are active sort columns and in which direction. Verified with Playwright.

**FR25:** After a page refresh, the Universe screen (and any other screen that persists sort/filter state via `x-table-state`) must visually restore the sort indicators in column headers and the active filter state in the filter UI (e.g., filter chips, input values). Currently the data is correctly restored but the UI rendering does not reflect it. Verify with Playwright.

**FR26:** Storybook must present a theme switcher that allows viewing any story in both the light and dark Material themes. Snapshot testing must capture a screenshot of each story in both themes as separate named screenshots. Verify with Playwright.

**FR27a:** The Screener screen must have all checkbox columns removed from the table UI and from the backend schema, API routes, and sorting logic.

**FR27b:** The "Update Universe" action on the Universe screen (and its backend endpoint) must be updated to no longer require all checkboxes to be checked before adding symbols to the universe. It should add all symbols currently returned by the screener query to the universe directly, without any checkbox precondition. Verify with Playwright.

**FR28:** The Storybook instance must include a meaningful set of stories for the `BaseTable` component — at minimum a Default story using mock data representative of the tables used in the app, and at least two variation stories (e.g., specific table configurations). All stories must render visibly and correctly in Storybook. Verify with Playwright and snapshot testing.

**FR29:** All virtual-scroll tables in the application (Universe, Screener, Account, Global Summary) must use a consistent, fixed row height so that the virtual-scroll buffer calculation (10-row buffer) is accurate. Janky/jumping scrolling caused by inconsistent row height estimates must be eliminated. The goal is to remove the janky/jumping scrolling more than the specific fixed height requirement. Verify with Playwright scroll test.

---

### Non-Functional Requirements

**NFR-21-Reliability:** Rate-limiting directives in prompts must not break when the agent executes multi-step tasks. The approach must prefer delay/wait-between-calls strategies over hard API-call-count limits that might abort a task mid-execution.

**NFR-22-Quality:** The 100% branch coverage requirement applies to all application source files. Files excluded from coverage (config files, spec files, etc.) remain excluded per the existing `vitest.config.ts` exclude list. The threshold must be enforced per-file or globally — globally is acceptable as a first step.

**NFR-23-Governance:** Changes to quality tool configurations must follow an approval-first process. The quality tooling (jscpd, eslint, prettier) is a shared team concern and must not be silently modified by automated agents.

**NFR-24-UX:** Multi-column sort must be operable via the existing column-header click interaction. Shift+click or a secondary UI affordance may be used for the second sort column; the chosen mechanism must be intuitive and not conflict with existing single-click sort behaviour.

**NFR-25-UX:** The restored sort/filter UI state must be visually indistinguishable from a state that was set by the user in that same session — no flash or deferred render.

**NFR-27-Cleanup:** All removed checkbox-related code (columns, migrations, services, APIs) must be cleaned up completely with no orphaned references remaining.

**NFR-29-Performance:** The fixed row height for virtual-scroll tables must match the actual rendered row height measured in the browser to avoid layout recalculation loops that cause the janking behaviour.

---

### Additional Requirements (from Architecture)

- **ADR-004 (ESLint flat config) applies to FR22:** Any changes to the vitest threshold configuration must be made in `vitest.config.ts` (root coverage thresholds). Per-project vitest configs in `apps/` projects must also be audited to ensure thresholds are consistent or deferred to the root.
- **SmartNgRX / signal-first pattern applies to FR24/FR25:** Multi-column sort state and restored UI state must be managed as Angular signals; no direct DOM mutations.
- **`x-table-state` header (existing pattern):** FR24 sort state and FR25 restored state both travel via `x-table-state` base64 JSON header. The multi-column sort state shape must extend (not replace) the existing shape so that previously saved single-column states are still loadable without error.
- **Playwright verification (MANDATORY) for FR24, FR25, FR26, FR27, FR28, FR29:** A fix/feature is not complete until a Playwright test confirms correct behaviour. Build/lint passing alone is insufficient.
- **FR23a duplication exposure:** Once the jscpd ignore paths are removed, the duplicate-check (pnpm dupcheck) will likely fail. The exposed duplicated files must be refactored to eliminate the duplication before the epic is closed.
- **Architecture document update (FR23b):** Use the `bmad-architect` skill to add the Quality Tool Governance section. The section must also document all past approved changes to quality tool configuration (as a change log).

## Epic 21: Add Rate-Limit Directives to Agent Prompts

**Goal:** Update the coding agent prompt files (`.github/instructions/`, `_bmad/` workflows, and any `copilot-instructions.md`) to include explicit instructions that prevent rate-limit exhaustion when the agent calls the GitHub Copilot API. The instructions must favour task completion over speed — the agent must pause between API calls rather than abort when a limit is approached.

**Approach:** Identify all prompt/instruction files that govern how the AI agent operates in this workspace. Add a "Rate Limits" section or inline directives that instruct the agent to: (a) never issue more than one Copilot API call per 2-minute window when the limit is approached, (b) insert explicit wait/pause steps between calls when processing multi-step tasks, and (c) prefer completing a task slowly over failing fast due to rate limits.

---

### Story 21.1: Audit and Update Agent Prompt Files for Rate-Limit Directives

As a developer,
I want all agent instruction files to include rate-limit guidance,
So that the AI agent completes long-running tasks without being interrupted by Copilot API rate limits.

**Acceptance Criteria:**

**Given** the workspace's `.github/instructions/` folder, `_bmad/` config files, and any `copilot-instructions.md`,
**When** I review each file for rate-limit guidance,
**Then** every file that governs agent behaviour includes a "Rate Limits" section (or equivalent inline instructions) that states:
- Do not call the Copilot API more than once every 2 minutes if rate limiting is detected.
- Insert a wait/pause step between API-dependent operations when processing multi-step tasks.
- Prefer slow task completion over aborting a task mid-execution due to rate limits.

**And** the instructions are written in a form that an LLM will follow (imperative, specific, not vague).

**And** running `pnpm all` still passes after the file changes (no build/lint impact since these are markdown files).

## Epic 24: Multi-Column Sorting on Universe Screen

**Goal:** Extend the Universe screen's table to support sorting by at least 2 simultaneous columns. The primary sort determines overall order; the secondary sort is a tiebreaker within groups sharing the same primary value. The sort state must be persisted in the existing `x-table-state` header mechanism (backwards-compatible with single-column saved states). Verified with Playwright.

**Approach:** Extend the table state model to hold an ordered array of `{ column, direction }` sort entries. Update the column-header click logic so that Shift+click adds/toggles a secondary sort column instead of replacing the primary. Update the backend sort logic to sort by the array in order. Update the UI to display active sort indicators for all active sort columns (with a numeric rank indicator if there are 2+).

**Dependencies:** Depends on Epic 25 (sort state restoration UI) being in scope — they share the sort state model. Epic 24 introduces the model; Epic 25 ensures the UI reads it on restore.

---

### Story 24.1: Extend Sort State Model for Multi-Column Sort

As a developer,
I want the sort state model to support an ordered list of sort columns,
So that the backend and frontend can represent, persist, and restore multi-column sort order.

**Acceptance Criteria:**

**Given** the existing `x-table-state` sort state shape (currently a single `{ column, direction }` object or equivalent),
**When** I extend it to an array `sortColumns: Array<{ column: string; direction: 'asc' | 'desc' }>`,
**Then** the existing single-column saved states are still loadable: a saved state with a single-column representation is automatically migrated to the array format on load (the single entry becomes `sortColumns[0]`).

**And** the model change is covered by unit tests: loading a legacy state, loading a multi-column state, serialising a multi-column state round-trips correctly.

**And** `pnpm all` passes.

---

### Story 24.2: Implement Multi-Column Sort in Backend Query

As a developer,
I want the backend Universe query to apply multiple sort columns in order,
So that rows are sorted by the primary column first and by the secondary column within tied primary values.

**Acceptance Criteria:**

**Given** an `x-table-state` header containing `sortColumns` with 2 entries (e.g., `[{ column: 'position', direction: 'desc' }, { column: 'symbol', direction: 'asc' }]`),
**When** the backend Universe route processes the request,
**Then** the query applies `ORDER BY position DESC, symbol ASC` (or equivalent Prisma `orderBy` array).

**And** unit tests cover: single-column sort (backwards-compatible), two-column sort, and empty sort (default ordering).

**And** `pnpm all` passes.

---

### Story 24.3: Implement Multi-Column Sort UI on Universe Screen

As a developer,
I want the Universe screen's column headers to support Shift+click to add a secondary sort column,
So that users can sort by two columns simultaneously.

**Acceptance Criteria:**

**Given** the Universe screen table is loaded with data,
**When** the user clicks a column header (primary sort — existing behaviour),
**Then** the table sorts by that column and the header shows the sort direction arrow (existing behaviour).

**Given** a column is already sorted,
**When** the user Shift+clicks a second column header,
**Then** the table sorts by the first column first and the second column as tiebreaker, and both column headers show sort direction indicators with a numeric rank superscript (e.g., "1↑" and "2↓").

**Given** the user Shift+clicks the currently-active secondary column header,
**When** the sort direction is `asc`,
**Then** the sort direction for that column toggles to `desc`.

**Given** the user clicks a column header without Shift,
**When** there are active secondary sort columns,
**Then** the secondary sort columns are cleared and only the clicked column is active (full sort reset to single column).

**And** the sort state is saved to `x-table-state` after each change.

**And** Playwright verifies: primary sort, secondary sort addition, rank indicators visible, sort state persisted across a hard reload.

## Epic 29: Fix Janky Virtual-Scroll Table Scrolling

**Goal:** Eliminate the jumpy/janky scrolling behaviour on all virtual-scroll tables (Universe, Screener, Account, Global Summary). The root cause hypothesis is inconsistent row heights — when `CdkVirtualScrollViewport` uses an incorrect `itemSize`, the 10-row buffer calculation is wrong and the browser's layout engine fights the virtual-scroll container, causing position jumps. Verified with Playwright scroll test.

**Approach:** Measure the actual rendered row height for each table. If all tables use the same row height, define a single shared constant. If they differ, set explicit `itemSize` values per table. Convert any row height values defined in pixels in CSS to Tailwind equivalents (consistent with existing Tailwind migration). See if fixing `itemSize` alone resolves the janking before investigating CDK viewport configuration.

**Dependencies:** Epic 5 (CSS → Tailwind migration) should be complete so that row heights are expressed as Tailwind utilities rather than raw CSS. However, this epic can proceed independently if needed — the primary fix is setting the correct `itemSize`, not migrating CSS.
