# DMS Workspace — Epic Breakdown (2026-03-30)

## Overview

This document provides the epic and story breakdown for new epics starting at Epic 30, based on the requirements defined on 2026-03-30. These epics focus on developer tooling reliability, virtual-scroll UX correctness, code quality enforcement, and Storybook display correctness.

---

## Requirements Inventory

### Functional Requirements

- FR1: All prompt files that contain the rate-limiting directive must use the new explicit wording covering terminal calls, MCP server calls, and code updates.
- FR2: The Universe screen table must scroll smoothly without the header jumping along with rows during incremental vertical scrolling.
- FR3: The Account screen table(s) must scroll smoothly without the header jumping along with rows during incremental vertical scrolling.
- FR4: The jscpd copy-paste detection threshold must be set to 0 (zero tolerance), not 0.1%.
- FR5: All copy-paste violations surfaced after lowering the threshold must be resolved by refactoring.
- FR6: The Architecture documentation must be updated to reflect the jscpd threshold of 0.
- FR7: The BaseTable Storybook story must render visibly by wrapping the component in a parent `<div>` with a fixed height and width.

### Non-Functional Requirements

- NFR1: Rate-limit directive changes must be consistent across every affected prompt/instruction file (no file left with the old wording).
- NFR2: Virtual-scroll table fix must not regress row rendering performance with ≥ 1,000 rows.
- NFR3: The Playwright MCP server must be used to reproduce the janky scroll issue and to verify the fix.
- NFR4: jscpd violations must be resolved by refactoring to shared utilities — suppressions are not allowed.
- NFR5: Storybook story fix must follow the existing Storybook ADR-001 patterns (applicationConfig, co-located stories file).

### Additional Requirements (Architecture)

- jscpd threshold: Architecture doc specifies 0.1% — must be updated to 0 to match the corrected `.jscpd.json` configuration.
- Rate-limit directives exist across 17 prompt/instruction files; all must be updated atomically.
- Virtual-scroll fix must be validated with Playwright MCP server before and after (reproduce → fix → verify).
- Storybook pattern: stories are co-located with component files under `apps/dms-material/src/app/`.

## Epic 30: Update Rate-Limiting Directives in Agent Prompts

Developers working with AI agents experience fewer rate-limit failures because all prompt and instruction files now use explicit, unambiguous language specifying terminal calls, MCP server calls, and code updates — rather than the vague term "API calls".

### Story 30.1: Replace Rate-Limiting Directive in All Prompt and Instruction Files

As a developer working with AI agents,
I want all prompt and instruction files to use the updated, explicit rate-limiting directive,
So that agents know precisely what actions require a wait and rate-limit failures are reduced.

**Acceptance Criteria:**

**Given** all files under `.github/prompts/` and `.github/instructions/` that contain the old rate-limiting statement,
**When** the replacement is applied,
**Then** every occurrence of:

```
- Always wait for at least 2 minutes between API calls.
```

is replaced with:

```
- Always wait for at least 2 minutes between terminal calls, calls to MCP servers, or code updates.
```

**And** the surrounding block (including "To avoid GitHub Copilot rate limiting:" and "Prefer slow completion over fast failure.") is preserved exactly as-is — only the single bullet line changes.

**And** `grep -r "between API calls" .github/` returns zero results.

**And** `grep -r "between terminal calls, calls to MCP servers, or code updates" .github/` returns the same count of results as the old directive had (19 occurrences across 17 files).

**And** `pnpm all` passes.

## Epic 31: Fix Janky Virtual-Scroll Table Header Jumping

Users can scroll the Universe and Account screen tables smoothly without the table header jumping or stuttering along with the rows, regardless of how many rows are loaded or how incrementally the user scrolls. This epic supersedes Epic 29, which did not fully resolve the issue.

### Story 31.1: Reproduce and Root-Cause the Header Jumping Issue

As a developer,
I want to use the Playwright MCP server to reproduce the janky header-jumping issue and identify its root cause,
So that the fix in Story 31.2 targets the correct problem.

**Acceptance Criteria:**

**Given** the application is running locally with the Universe screen loaded with at least 50 rows,
**When** I use the Playwright MCP server to scroll the virtual-scroll table incrementally (small increments, not a single jump to the bottom),
**Then** the Playwright MCP session confirms the header row moves vertically along with the data rows during the scroll — documenting the exact DOM/CSS behaviour causing the jump (e.g. transform values on the viewport content container, sticky positioning failure, etc.).

**And** the root cause is documented in `_bmad-output/implementation-artifacts/virtual-scroll-header-fix.md`, including:

- The CSS property or DOM mechanism causing the header to jump
- Which Angular Material / CDK version constraint applies
- The proposed fix approach

**And** the same behaviour (or its absence) on the Account screen is also documented.

---

### Story 31.2: Implement and Verify the Virtual-Scroll Header Fix

As a user,
I want the table header to remain stationary as I incrementally scroll through a large table,
So that I can always see column labels while reading rows.

**Acceptance Criteria:**

**Given** the root-cause documented in Story 31.1,
**When** the fix is implemented in `BaseTableComponent` (and any relevant SCSS or host CSS),
**Then** the Playwright MCP server can incrementally scroll the Universe screen table and the header row does not move vertically at any point — its `getBoundingClientRect().top` value stays constant throughout the scroll.

**And** the same verification passes on the Account screen table (open positions, sold positions, and dividend deposits tabs).

**And** no existing Playwright E2E tests are broken.

**And** virtual-scroll performance is not regressed: scrolling through 1,000 rows remains fluid (no visible frame drops).

**And** `pnpm all` passes.

## Epic 32: Fix Copy Paste Detection Threshold and Violations

The project enforces zero-tolerance copy-paste detection (`.jscpd.json` threshold = `"0"`), all duplications surfaced by the stricter threshold are eliminated by refactoring, and the Architecture documentation reflects the corrected standard. No suppression entries are added.

### Story 32.1: Lower jscpd Threshold to Zero and Update Architecture Doc

As a developer maintaining code quality,
I want the jscpd copy-paste threshold set to zero tolerance,
So that no code duplication is permitted to accumulate undetected.

**Acceptance Criteria:**

**Given** `.jscpd.json` currently has `"threshold": "0.1"`,
**When** the change is applied,
**Then** `.jscpd.json` has `"threshold": "0"`.

**And** `_bmad-output/planning-artifacts/architecture.md` is updated so the sentence "**Threshold:** 0.1% maximum duplication across the scanned codebase" reads "**Threshold:** 0% — zero tolerance for code duplication across the scanned codebase".

**And** no entries are added to the `ignore` array.

**And** `pnpm dupcheck` is run and its output (pass or list of violations) is captured — if violations exist, they are recorded in `_bmad-output/implementation-artifacts/jscpd-violations.md` for Story 32.2.

**And** if `pnpm dupcheck` already passes at threshold 0, `pnpm all` passes and this epic is done (Story 32.2 is a no-op).

---

### Story 32.2: Refactor All Copy-Paste Violations to Pass Zero-Threshold Check

As a developer maintaining code quality,
I want all code duplications identified by jscpd at threshold 0 to be eliminated,
So that `pnpm dupcheck` passes with zero violations.

**Acceptance Criteria:**

**Given** the violations documented in Story 32.1 (or discovered by running `pnpm dupcheck` at threshold 0),
**When** each violation is resolved,
**Then** duplicated logic is extracted into a shared utility, service, or base abstraction following the `@smarttools/one-exported-item-per-file` rule — no suppression entries are added to `.jscpd.json`.

**And** each extracted shared file has a corresponding spec file if it contains non-trivial logic.

**And** `pnpm dupcheck` reports zero violations.

**And** `pnpm all` passes (lint, build, test, dupcheck, format all green).

## Epic 33: Fix BaseTable Storybook Story Display

Developers opening the BaseTable stories in Storybook see the component rendered visibly because the story wraps the component in a host `<div>` with explicit fixed height and width, matching the ADR-001 Storybook patterns.

### Story 33.1: Add Fixed-Size Host Wrapper to BaseTable Storybook Stories

As a developer,
I want the BaseTable Storybook stories to render the component visibly with a realistic layout,
So that I can inspect and test the table's appearance and behaviour in isolation.

**Acceptance Criteria:**

**Given** the current `base-table.stories.ts` where the `Default`, `EmptyState`, and `UniverseTableVariation` stories render with zero or unconstrained height,
**When** the fix is applied,
**Then** each story's `render` (or `decorators`) wraps the `<dms-base-table>` component element in a host `<div>` styled with `height: 500px; width: 100%; display: block;` (or equivalent explicit dimensions) so the virtual-scroll viewport has a bounded container to render into.

**And** opening Storybook (`pnpm storybook`) and navigating to `Shared/BaseTable` shows the `Default` story with a visible table (header row and data rows rendered).

**And** the `EmptyState` story renders the "no rows" state visibly within the bounded container.

**And** the `UniverseTableVariation` story renders the full universe-column table visibly.

**And** the fix uses the `applicationConfig` / `decorators` pattern from ADR-001 — no TestBed patterns.

**And** `pnpm storybook:build` succeeds without errors.

**And** `pnpm all` passes.
