# dms-workspace — Epic Breakdown (2026-05-05)

## Overview

Two new epics derived from the 2026-05-05 epic descriptions file. These epics extend the
project beyond the current backlog (Epics 1–96) starting at **Epic 97**.

- **Epic 97 — Restore Missing Open Positions Computed Fields on the Server:** The Open Positions
  tab is missing computations for **Expected$**, **Last$ Unrlz Gain%**, **Unrlz Gain$**, and
  **Target Gain**. These were previously computed client-side and were inadvertently removed when
  symbol resolution moved server-side (the same architectural shift addressed by Epics 94–96).
  This epic re-implements the four missing fields on the server (so `Trade` rows arrive at the
  client with values populated), and moves the existing client-side **Target Sell** computation
  to the server as well, leaving the client purely presentational.

- **Epic 98 — Electron Setup Scripts (Database and Per-Platform Builds):** Three independent
  concerns required to make the packaged Electron app installable and maintainable:
  1. **First-install database creation** at `~/.dms/dms.db` (Linux/macOS) and
     `C:\Users\USERNAME\.dms\dms.db` (Windows). This refines the `userData`-based path
     researched in Epic 91 by standardising on a `~/.dms/` directory across platforms.
  2. **Node-free database migrations.** The packaged Electron app cannot rely on Node being
     installed on the user's machine, so Prisma migrations cannot be invoked the standard way.
     The preferred approach is to embed Prisma's migration engine binary inside the asar
     resources; the fallback is to create a fresh DB at the new schema and copy data across
     from the old one.
  3. **Per-platform build splitting.** The current build attempts Linux, macOS, and Windows
     in a single pipeline; macOS must be built on a Mac and currently fails (no Mac
     available), which blocks the Linux and Windows builds. Splitting the build by platform
     allows each target to be built (and to fail) independently.

All stories follow existing project conventions:

- **Angular 21 zoneless** — `inject()`, `OnPush`, signal-first
- **SmartNgRX / SmartSignals** for state management
- **Fastify** backend, **Prisma** ORM (SQLite dev / SQLite-on-disk in Electron)
- **Vitest** for unit tests, **Playwright** for E2E
- **Tests are authoritative** — do not change tests unless a test is asserting something we
  are explicitly changing
- **`pnpm all`** must pass after every story
- **Playwright MCP server** must be used to reproduce and verify all UI failures
- **TDD** — failing test first, then implementation, then regression-prevention coverage

## Epic 97 — Restore Missing Open Positions Computed Fields on the Server

**Goal:** Restore the four open-positions computed fields (Expected$, Last$ Unrlz Gain%,
Unrlz Gain$, Target Gain) that were lost when symbol resolution moved server-side, and move
the remaining client-side Target Sell computation to the server. After this epic, the
`open-positions-component.service.ts` performs no per-row arithmetic — it only renders the
values supplied by the server.

**Scope:** Server `Trade` mapping in `apps/server/src/app/routes/trades/index.ts`, the
client `Trade` interface in `apps/dms-material/src/app/store/trades/trade.interface.ts`, the
`open-positions-component.service.ts` (remove computations), and the matching e2e tests.

**Out of scope:** Sold Positions and Dividend Deposits computed fields (covered by Epics 94
and 96 for symbol resolution; any sold-positions computed-field gaps are deferred).

## Epic 98 — Electron Setup Scripts (Database and Per-Platform Builds)

**Goal:** Make the packaged Electron app installable and updatable on a clean machine (no
Node toolchain assumed) and decouple the per-platform build pipelines so a macOS failure
does not block Linux or Windows artifacts.

**Scope:** `apps/electron/` (main process, packaging configuration), `prisma/` (migration
bundle layout), and the build/CI scripts that drive `electron-builder`. Builds on the Epic 91
research foundation but standardises the database location at `~/.dms/dms.db` instead of the
platform-specific `userData` directory.

**Out of scope:** Auto-update mechanism (delivery of new app versions); only the
on-disk-database migration step is covered here.

---

### Story 98.1 — First-Install Database Creation at `~/.dms/dms.db`

**Status:** Approved

**As Dave (the investor),**
**I want** the Electron app to create `~/.dms/dms.db` (Linux/macOS) or
`C:\Users\<USERNAME>\.dms\dms.db` (Windows) on first launch if it does not yet exist,
**So that** I can install the app and have it work immediately without any manual database
setup.

**Acceptance Criteria:**

**Given** a clean machine where `~/.dms/dms.db` does not exist,
**When** the packaged Electron app is launched for the first time,
**Then** the Electron main process creates the `~/.dms/` directory (mode `0o700` on POSIX)
if missing and creates an empty SQLite file at `~/.dms/dms.db`.

**Given** the database path is resolved cross-platform,
**When** the path is computed,
**Then** it uses Node's `os.homedir()` joined with `.dms/dms.db` (this resolves correctly
to `C:\Users\<USERNAME>\.dms\dms.db` on Windows without per-platform branching).

**Given** the database file already exists at the path,
**When** the app is launched,
**Then** the main process leaves the existing file untouched and proceeds to the migration
step (Story 98.2).

**Given** the database path is established,
**When** the Electron main process forks the Fastify server,
**Then** it sets `process.env['DATABASE_URL'] = `file:${dbPath}`` so the forked server
process inherits the correct location.

**Given** TDD ordering,
**When** the unit tests for the path-resolution and create-if-missing helpers are written,
**Then** they fail before the implementation exists and pass after, and `pnpm all` passes.

**Definition of Done:**

- [ ] `apps/electron/src/main.ts` (or a new helper) computes the path from `os.homedir()`
- [ ] `~/.dms/` is created with mode `0o700` on POSIX if missing
- [ ] Empty `dms.db` file is created if missing; existing file is left untouched
- [ ] `DATABASE_URL` is set before the server is forked
- [ ] Unit tests cover the helper (POSIX and Windows paths)
- [ ] `pnpm all` passes
- [ ] `pnpm format` passes

### Story 98.2 — Node-Free Prisma Migrations on Launch

**Status:** Approved

**As Dave (the investor),**
**I want** the packaged Electron app to apply pending Prisma migrations to `~/.dms/dms.db`
on launch without requiring Node, npm, or pnpm to be installed on my machine,
**So that** I can install a new app version and have my database schema upgraded
automatically.

**Depends on:** Story 98.1

**Acceptance Criteria:**

**Given** the packaged app contains the Prisma migration engine binary and migration SQL
files in `process.resourcesPath` (outside the asar),
**When** the Electron main process launches,
**Then** it invokes the bundled migration engine against `~/.dms/dms.db` to apply any
pending migrations before forking the Fastify server.

**Given** no Node binary is on the user's PATH,
**When** the migration runs,
**Then** it uses only the bundled binary (and any DLL/SO it depends on) — no `node`,
`npm`, or `pnpm` invocation occurs.

**Given** the embedded-migration-engine approach is investigated and documented,
**When** the developer determines it is infeasible,
**Then** the fallback path is implemented: create a fresh `dms.db.new` at the new schema
using bundled migration SQL, copy rows from the old DB to the new DB using a SQLite-only
script, then atomically replace `dms.db` with `dms.db.new`. The chosen path is documented
in Dev Notes.

**Given** a database that is already at the latest schema,
**When** the migration step runs,
**Then** it is a no-op (no migrations applied) and the server starts normally.

**Given** the migration step fails,
**When** the failure occurs,
**Then** the Electron main process surfaces a clear error dialog (does not silently
proceed with a stale schema) and exits non-zero.

**Given** all changes,
**When** `pnpm all` runs,
**Then** all tests pass, including a unit test that mocks the migration invocation to
verify the main process calls it before forking the server.

**Definition of Done:**

- [ ] Prisma migration engine binary bundled in `process.resourcesPath` (outside asar)
- [ ] Migration SQL files bundled alongside the binary
- [ ] Main-process code invokes the binary on launch, before the server fork
- [ ] No-op behaviour verified when DB is up-to-date
- [ ] Failure surfaces an Electron error dialog and exits non-zero
- [ ] Approach (embedded engine vs fallback copy) documented in Dev Notes
- [ ] Unit tests pass; `pnpm all` passes
- [ ] `pnpm format` passes
