# dms-workspace — Epic Breakdown (2026-04-10)

## Overview

Five new epics derived from the 2026-04-10 epic descriptions file. This batch extends the project
backlog starting at **Epic 62**. The work covers: a second attempt at ensuring newly-added symbols
retrieve sufficient distribution history to correctly classify payment frequency (Epic 58 fixed the
algorithm but the regression persists); a targeted fix for the recurring "No open lots for
\${symbol}" split-import error (Epics 48, 57, 61 have attempted fixes — the root cause is that the
CSV processor encounters split events before buy transactions when the input rows are not being

- **Angular 21 zoneless** — `inject()`, `OnPush`, signal-first
- **SmartNgRX / SmartSignals** for state management and lazy loading
- **Vitest** for unit tests, **Playwright** for e2e
- **Tests are authoritative** — do not change tests unless a test is asserting something we are
  explicitly changing

### Additional Requirements (Architecture / Context)

- **Distribution frequency — Epic 62:** Epic 58 extended `calculateDistributionsPerYear` (in
  `apps/server/src/app/routes/settings/common/get-distributions.function.ts`) to handle the sparse-
  history case. Despite that fix, `distributions_per_year` still defaults to 1 for some newly-added
  symbols. Story 62.1 must read the current production implementation of the fixed function, run the
  existing failing unit test from Story 58.1 (to confirm it passes or still fails), and then use the
  Playwright MCP server to add a live monthly or weekly payer and observe the stored
  `distributions_per_year` value. Only after the regression mode is confirmed may Story 62.2 attempt
  a fix. The fix must cover all four cadences and must not regress the annual / quarterly paths that
  were working before Epic 58.

- **CSV processing order — Epic 63:** The Fidelity CSV export is ordered in reverse chronological
  order **within each account group** (most-recent transaction first). For a reverse split, the
  split event row has a later date than the preceding buy rows — so if the pipeline walks the CSV
  **forwards** (index 0 → end), it encounters the split before the buys, finds no open lots, and
  emits the "No open lots for \${symbol}" error. The fix is to either sort or reverse-iterate the
  rows **before** the processing loop so that buy transactions are always inserted before split
  events are attempted. Story 63.1 must write a failing E2E test using a minimal reproducer CSV that
  exhibits the error. Story 63.2 fixes the ordering. Story 63.3 adds regression coverage.

- **Janky scrolling — Epic 64:** The janky scrolling defect has been fixed four times (Epics 29,
  31, 44, 60) and regresses each time. The last known fix (Epic 60) addressed an `isLoading` state
  emission race from SmartSignals that caused rows entering the viewport to momentarily return a
  null/loading row before hydrating. If the defect has returned yet again, Story 64.1 must use the
  Playwright MCP server to confirm the exact current failure mode before any code is touched. The
  fix in Story 64.2 must include explanatory comments referring to Epics 29, 31, 44, and 60 so
  future maintainers understand the full history.

- **Unloaded symbols on deep scroll — Epic 65:** This is a different manifestation from Epic 60:
  blank rows appear at the **end** of the sorted list, not the beginning, and only after the user
  has scrolled down enough to trigger multiple lazy-load batches. The symptom suggests that rows
  loaded in batch N are not receiving their signal updates when the viewport scrolls into batch N+1
  (signals for earlier batches may not be re-emitting once the SmartNgRX entity is already
  partially stored). Story 65.1 must reproduce the exact symptom using the Playwright MCP server by
  scrolling past at least two lazy-load page boundaries before asserting. The fix in Story 65.2
  targets the SmartSignals / SmartNgRX layer that handles multi-batch entity hydration.

- **CEF classification on symbol add — Epic 66:** The screener pipeline already contains code that
  queries CefConnect.com and classifies a symbol as Equity, Income, or Tax Free. The add-symbol
  flows (`apps/server/src/app/routes/universe/add-symbol/`) currently assume every new symbol is an
  equity and do not perform a CefConnect lookup. Story 66.1 refactors the screener's classification
  logic into standalone functions (if not already functions) so they can be called from the add-
  symbol path. Stories 66.2 and 66.3 wire these functions into the CSV import and + button flows
  respectively. Story 66.4 adds E2E coverage ensuring that a known CEF symbol is classified
  correctly when added via both paths.

## Epic 63: CSV Split Processing Order

**Goal:** The CSV import pipeline has tried to handle reverse splits in Epics 48, 57, and 61 — the
CUSIP-to-ticker lot-resolution bug was addressed in Epic 61, but a separate ordering problem
remains: the pipeline encounters split event rows **before** the corresponding buy rows because the
Fidelity CSV export is in reverse chronological order and the pipeline walks rows front-to-back.
When the split processor runs first it finds no open lots and emits "No open lots for \${symbol}",
aborting the adjustment. The fix is to walk the CSV in reverse date order (or sort rows ascending
by date before processing) so that all buy transactions for a symbol are inserted into the database
before any split event for that symbol is attempted.

## Epic 66: CEF Classification on Symbol Add

**Goal:** When a new symbol is added to the universe — either through CSV import or the manual +
button — the application currently defaults to classifying it as an equity without querying
CefConnect.com. Because a significant portion of income-focused symbols are Closed End Funds (CEFs)
that must be classified as Equity, Income, or Tax Free, this default causes the annualised yield
and sector grouping to be wrong from the first day the symbol appears. The screener pipeline
already contains working CefConnect integration and classification logic. This epic refactors that
logic into reusable functions and wires them into both add-symbol code paths so that CEF
classification happens automatically on every new symbol add.
