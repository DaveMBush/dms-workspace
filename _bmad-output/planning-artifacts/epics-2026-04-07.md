Four new epics derived from the 2026-04-07 epic descriptions file. This batch extends the project
backlog (Epics 1–53) starting at **Epic 54**. The work covers: restoring the Universe screen sort

### Additional Requirements (Architecture / Context)

- **Sort stickiness (Epic 54):** State is stored in `localStorage` under key `'dms-sort-filter-state'`
  via `SortFilterStateService`. `loadSortColumnsState('universes')` is called in
  `GlobalUniverseComponent.ngOnInit`. Epic 43 changed the secondary-sort behaviour; the regression
  likely involves either the save path (`saveSortColumnsState`) not being called after the sort
  change, or the load path restoring state before the SmartNgRX effect service is ready.  The fix
  must not break the multi-column sort introduced in Epic 43.
- **Duplicate symbols (Epic 55):** The backend `GET/POST /api/universe` endpoint returns rows
  computed from `trades` via Prisma. Duplicate rows likely originate from the Prisma query fan-out
  on the `trades` relation or a missing `DISTINCT` / `GROUP BY` when the sort involves a computed
  field.  The `parseSortFilterHeader` utility applies the active account filter via request headers.
- **Empty rows on Symbol sort (Epic 56):** Universe rows are loaded lazily via SmartNgRX
  `loadByIds`. When sort is by a column that places rows with index-based IDs at the top the
  virtual-scroll viewport may request IDs that SmartSignals treats as "already in the store but
  not yet hydrated", causing the rendered rows to be empty until a re-scroll triggers a second
  load.  The fix should target the effect service or the component's virtual-scroll initialisation,
  not the sort logic itself.
- **Reverse-split import (Epic 57):** `isSplitRow()`, `calculateSplitRatio()`, and
  `adjustLotsForSplit()` already exist in `apps/server/src/app/routes/import/` and a basic e2e
  test (`split-import-e2e.spec.ts`) exists for OXLC. The existing code may not handle the
  CUSIP-based "R/S FROM {cusip}#REOR" pattern (where the "FROM" row carries the new symbol and the
  "TO" row carries the old CUSIP as the symbol). Story 57.1 must investigate what is and is not
  handled before any code is written.

### Story 57.2: Implement Correct Reverse-Split Detection and Ratio Calculation

**Slug:** `57-2-implement-reverse-split-detection-ratio`

As a portfolio manager,
I want the CSV importer to detect and correctly calculate the ratio for Fidelity reverse-split
row pairs,
so that my open lots are adjusted with the right multiplier.

**Acceptance Criteria:**

**Given** a CSV containing a "REVERSE SPLIT R/S FROM {cusip}#REOR" row followed by the matching
"REVERSE SPLIT R/S TO {cusip}#REOR" row,
**When** the import processes the file,
**Then** the pair is recognised as a reverse split and the ratio is derived as |TO row quantity|
÷ FROM row quantity (e.g. 400 ÷ 80 = 5 for MSTY).

**Given** a MSTY reverse-split pair (1-for-5) is imported,
**When** the import completes,
**Then** each open MSTY lot has its quantity divided by 5 (floored) and its buy price multiplied
by 5.

**Given** a ULTY reverse-split pair (1-for-10) is imported,
**When** the import completes,
**Then** each open ULTY lot has its quantity divided by 10 (floored) and its buy price multiplied
by 10.

**Given** a OXLC reverse-split pair (1-for-5) is imported,
**When** the import completes,
**Then** each open OXLC lot has its quantity divided by 5 (floored) and its buy price multiplied
by 5.

**Given** the CUSIP on the "R/S TO" row does not match any known symbol,
**When** the import attempts the lot adjustment,
**Then** the symbol is resolved from the trades table by CUSIP before adjusting lots (or the pair
is skipped with a warning if resolution fails).

**Given** fractional-share remainders exist after the split,
**When** the import processes the remainder,
**Then** they are handled consistently with the existing `adjustLotsForSplit` contract (aggregated
fractional sale at market price).

**Given** all changes are applied,
**When** `pnpm all` runs,
**Then** all unit tests pass with no regressions.

**Definition of Done:**

- [ ] `isSplitRow()` (or a new detection function) correctly identifies "REVERSE SPLIT R/S FROM" rows
- [ ] Ratio is derived from the paired row quantities (not from description text alone)
- [ ] `adjustLotsForSplit()` is called with the correct ratio and account scope for MSTY, ULTY, and OXLC examples
- [ ] CUSIP-to-symbol resolution is applied when the "TO" row symbol is a raw CUSIP
- [ ] Unit tests added/updated in `apps/server/src/app/routes/import/` for the new detection and ratio logic
