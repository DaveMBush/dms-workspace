# dms-workspace — Epic Breakdown (2026-04-29)

## Overview

Three new epics derived from the 2026-04-29 epic descriptions file.

- **Epic 89 — CEF Set to Expired on Add:** When a new symbol is confirmed as a closed-end fund
  (`is_closed_end_fund: true`) during a manual add or CSV import, the `expired` flag must also
  be set to `true` at the moment of creation. CEFs that are added manually are, by definition,
  not currently present in the screener, so flagging them expired immediately allows the UI
  "expired CEF" filter to work correctly from day one.

- **Epic 90 — Distribution Frequency Normalization for Volatility:** The volatility engine
  (`calculateVolatility`) receives a flat array of raw per-payment distribution amounts and
  treats each element as a single comparable data point. When a symbol's payment cadence
  changes (e.g., monthly → quarterly), the per-payment amount jumps (e.g., $0.10 → $0.30)
  without any real change in the annualized distribution rate, producing a false volatility
  spike. This epic normalizes each raw amount to its monthly-equivalent before feeding it to
  `calculateVolatility`, so the function always compares apples to apples across cadence changes.

- **Epic 91 — Electron Installation Script:** The existing Electron app (`apps/electron`)
  runs only in development mode and requires Node.js and project dependencies to be installed.
  This epic creates a cross-platform packaging pipeline (using `electron-builder`) that
  produces self-contained installers for Linux, macOS, and Windows — bundling the Electron
  shell, the compiled Fastify server, the built Angular app, and the SQLite database together
  so the application can be installed and run as a native desktop app without any development
  toolchain. The packaging step also ensures the Prisma schema migration runs automatically
  the first time the installed app launches, keeping the database schema current across upgrades.

All stories follow existing project conventions:

- **Angular 21 zoneless** — `inject()`, `OnPush`, signal-first
- **SmartNgRX / SmartSignals** for state management
- **Fastify** backend, **Prisma** ORM (SQLite dev / PostgreSQL prod)
- **Vitest** for unit tests, **Playwright** for E2E
- **Tests are authoritative** — do not change tests unless a test is asserting something we are
  explicitly changing
- **`pnpm all`** must pass after every story
- **Playwright MCP server** must be used to reproduce and verify all UI failures
- **TDD** — failing test first, then implementation, then regression-prevention coverage

  `recalculate-universe-volatility.function.ts`. The minimum-data-points threshold (`MIN_DATA_POINTS
  = 12`) inside `calculateVolatility` counts normalized monthly-equivalent data points — so the

## Epic 90: Distribution Frequency Normalization for Volatility

**Goal:** Prevent false volatility spikes caused by a symbol switching its distribution cadence
(e.g., monthly → quarterly). Before feeding raw distribution amounts to `calculateVolatility()`,
normalize each amount to its monthly-equivalent so the function always receives a
frequency-agnostic series.

---

### Story 90.1: Implement Distribution Frequency Normalization Helper

**Slug:** `90-1-implement-distribution-frequency-normalization`

As a developer,
I want a `normalizeToMonthlyEquivalents(history: ProcessedRow[]): number[]` helper inside
`recalculate-universe-volatility.function.ts` that converts each raw per-payment amount to
its monthly equivalent by inferring the payment interval from consecutive row dates,
So that `calculateVolatility()` receives a comparable series regardless of whether the symbol
pays monthly, quarterly, or annually.

**Acceptance Criteria:**

1. **Given** a `ProcessedRow[]` where all payments are monthly (consecutive dates ~30 days apart),
   **When** `normalizeToMonthlyEquivalents` is called,
   **Then** each output amount equals the input amount (multiplier ≈ 1).

2. **Given** a `ProcessedRow[]` where all payments are quarterly (consecutive dates ~90 days apart),
   **When** `normalizeToMonthlyEquivalents` is called,
   **Then** each output amount equals `inputAmount / 3` (quarterly → monthly equivalent).

3. **Given** a `ProcessedRow[]` where payments switch mid-history from monthly to quarterly,
   **When** `normalizeToMonthlyEquivalents` is called,
   **Then** each amount is independently normalized based on its own inferred interval, so the
   series shows no artificial spike at the cadence-change boundary.

4. **Given** a `ProcessedRow[]` with a single row (no previous date to infer from),
   **When** `normalizeToMonthlyEquivalents` is called,
   **Then** the single amount is returned unchanged (multiplier = 1, safest default).

5. **Given** the change is implemented with failing unit tests first (TDD),
   **When** `pnpm all` runs,
   **Then** all tests pass, including the new spec for `normalizeToMonthlyEquivalents`.

**Definition of Done:**

- [ ] Failing unit tests written before implementation covering at least: all-monthly,
      all-quarterly, mixed-cadence, and single-row inputs
- [ ] `normalizeToMonthlyEquivalents` function implemented inside
      `recalculate-universe-volatility.function.ts` (not exported — internal helper)
- [ ] `calculateVolatility()` is NOT modified
- [ ] `pnpm all` passes
