# dms-workspace — Epic Breakdown (2026-04-13)

## Overview

Six new epics derived from the 2026-04-13 epic descriptions file. This batch extends the project
backlog starting at **Epic 67**. The work covers: normalising inconsistent row heights so that CDK
virtual scrolling always receives the correct `itemSize` (Epic 67); removing the misfiring "No
results found" message from the Universe screen which renders above column headers (Epic 68); fixing
a regression introduced by Epics 63–66 that causes Fidelity CSV imports to return a 400 Bad Request
(Epic 69); restoring the broken Error Logs route `global/error-logs` whose Angular route definition
was dropped (Epic 70); diagnosing why the screener-to-universe sync path leaves `distributions_per_year`
at 1 for many symbols when it should be 12 — a separate code path from the Epic 62 fix (Epic 71);
and forcing the Universe table to re-sort in place after any in-cell value update so the row
immediately appears at its correct sorted position without changing scroll position (Epic 72).

All stories follow the existing project conventions:

- **Angular 21 zoneless** — `inject()`, `OnPush`, signal-first
- **SmartNgRX / SmartSignals** for state management and lazy loading
- **Vitest** for unit tests, **Playwright** for e2e
- **Tests are authoritative** — do not change tests unless a test is asserting something we are
  explicitly changing
- **`pnpm all`** must pass after every story
- **Playwright MCP server** must be used to reproduce and verify all UI bugs
- Stories follow pattern: diagnose/reproduce → fix → e2e verification (for bug fixes)

### Additional Requirements (Architecture / Context)

- **Row height consistency — Epic 67:** The audit in `docs/row-height-audit.md` (Story 29.1)
  established that Angular Material renders data rows at 52px via
  `--mat-table-row-item-container-height`. The `BaseTableComponent` default `rowHeight` input is
  52. However, rows that contain an `<button mat-icon-button>` (which adds padding/min-height to
  the cell) are rendering larger than rows without them, producing a mismatch that misleads the CDK
  virtual scroll viewport. The fix must ensure all rows are visually identical — likely by pinning
  the Material token to the chosen height in SCSS — and that `rowHeight` in
  `base-table.component.ts` matches. Any e2e spec that hardcodes `52` or `57` as a pixel constant
  (e.g. `ROW_HEIGHT_PX = 52` in `universe-lazy-load-deep-scroll.spec.ts`) must be updated to match
  the new chosen height.

- **"No results found" message — Epic 68:** The message lives in
  `global-universe.component.html` lines 82–90 inside an `@if (showEmptyState$())` block. The
  `showEmptyState$()` computed in `global-universe.component.ts` returns `true` when
  `globalLoading.isLoading() && filteredData$().length === 0`. During page load, this produces a
  flash of the message above column headers. The fix is simply to remove the block. The
  corresponding unit test in `global-universe.component.spec.ts` that asserts `showEmptyState$` /
  `.empty-state` must be updated or removed along with the production code it was testing.

- **CSV Import 400 — Epic 69:** The import route `POST /api/import/fidelity` in
  `apps/server/src/app/routes/import/index.ts` handles both multipart file upload and plain-text
  body. The regression from Epics 63–66 must be diagnosed by reading the current import pipeline
  and comparing to what it was before those epics. The server log shows the request is received but
  no further logs appear, suggesting the handler throws before emitting a structured error. A
  synthetic fixture CSV modelled on `fidelity-valid.csv` should be used for e2e — do NOT embed
  paths like `/home/dave/Fidelity-2025.csv` in any code or test.

- **Error Logs route — Epic 70:** The Angular route configuration in
  `apps/dms-material/src/app/app.routes.ts` must include a path entry for `global/error-logs` that
  lazy-loads `apps/dms-material/src/app/global/global-error-logs/global-error-logs.ts`. The
  existing accessibility e2e test in `apps/dms-material-e2e/src/accessibility.spec.ts` (line 142)
  already navigates to `/global/error-logs` — if it was passing before, the route entry was
  present; if it is now failing, the route entry was removed. Story 70.1 must add the route entry
  back and confirm existing e2e tests pass.

- **Screener-to-Universe distribution history — Epic 71:** The sync-from-screener flow is in
  `apps/server/src/app/routes/universe/sync-from-screener/index.ts`. It calls `getDistributions(symbol)`
  and stores the result. The mock in `sync-from-screener/index.spec.ts` stubs `getDistributions` to
  return `distributions_per_year: 12`. If the production value is only 1, either: (a)
  `getDistributions` itself is returning 1 for these symbols (the Epic 62 / Epic 71 root cause), or
  (b) the sync path is not calling `getDistributions` for existing universe records on update.
  Story 71.1 must trace whether `updateExistingUniverseRecord` calls `getDistributions` or
  preserves the existing `distributions_per_year`. This is distinct from Epic 62, which was about
  newly-added symbols; Epic 71 is about the update path for symbols already in the universe.

- **Re-sort on row update — Epic 72:** The `handleCellEdit` function in
  `global-universe/handle-cell-edit.function.ts` mutates the row in-place and emits a `cellEdit`
  output event. The table's `filteredData$()` is a `computed()` signal that applies filters and
  sort. Because the mutation happens directly on the object reference (not via a new signal value),
  Angular's signal system may not detect the change as a reason to re-sort. The fix likely requires
  triggering a signal update after the edit (e.g. re-emitting the `sortColumns$` signal or
  explicitly refreshing `filteredData$`). The re-sort must preserve the current scroll position —
  use `viewport()?.scrollToIndex()` with the saved pre-edit scroll offset if needed.

## Epic 67: Inconsistent Row Heights

**Goal:** Rows that contain icon buttons are rendering taller (approximately 57px) than rows without
them (52px). The CDK virtual-scroll viewport receives a single `itemSize` value and assumes all
rows are the same height. Any height mismatch causes misaligned scroll positions, duplicated visible
rows, and jump artefacts — the same class of defect fixed repeatedly in Epics 29, 31, 44, and 60.
This epic pins all data rows to a single consistent height and updates `rowHeight` and any
pixel-hardcoded test constants to match.
