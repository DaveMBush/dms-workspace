# dms-workspace — Epic Breakdown (2026-04-28)

## Overview

One new epic derived from the 2026-04-28 epic descriptions file. This batch adds **Epic 88**, a
correction to the volatility data source introduced in Epic 81 and stored in Epic 85.

- **Epic 88 — Volatility From Distribution History (All Universe Symbols):** The current
  short-term and long-term dividend volatility calculation reads from the `divDeposits` table
  (broker-deposit history). That table only contains rows for symbols we have held positions
  for, and it has gaps if a symbol was sold and re-bought. As a result, volatility icons are
  blank or wrong for the majority of universe symbols — exactly the symbols where volatility is
  most useful for buy/sell decisions. The fix: when distribution history is fetched from
  `dividendhistory.net` during universe data retrieval, use that authoritative per-share
  distribution history as the input to the existing `calculateVolatility()` function and persist
  the result to the `universe.volatility_long` / `universe.volatility_short` columns added in
  Epic 85. This gives volatility for **every** symbol in the universe, not just those with a
  position history.

This epic supersedes the data-source aspect of Epics 81 and 85 (the calculation algorithm and
storage columns introduced there are reused unchanged — only the input data source changes). It
complements Epic 84, which fixed the rendering filter that hid icons for unheld symbols.

### Additional Requirements (Architecture / Context)

- **Distribution history is already fetched at universe-sync time.** The service
  `apps/server/src/app/routes/common/dividend-history.service.ts` exposes `fetchDividendHistory(symbol)`
  which returns the parsed dividendhistory.net rows. The universe sync flow already calls this
  per symbol when retrieving universe data — Story 88.1 must inventory the exact call sites and
  pick the integration point that gives volatility recalculation the same per-symbol distribution
  payload without re-fetching.

- **Calculation algorithm is unchanged.** The pure function in
  `apps/server/src/app/volatility/volatility-calculation.function.ts` (used by both
  `recalculate-universe-volatility.function.ts` and `volatility-query.function.ts`) is reused
  as-is. The only change is **what amounts array is fed in** — distribution-history per-share
  amounts instead of `divDeposits.amount` rows.

- **Storage columns already exist.** Epic 85 added `volatility_long`, `volatility_short`, and
  `volatility_calculated_at` to the `universe` Prisma model in both `schema.prisma` and
  `schema.postgresql.prisma`. No new migration is required for this epic.

- **Replacement, not toggle.** Per R36, the `divDeposits`-based path is removed. The
  `prisma.divDeposits.findMany(...)` calls inside `recalculate-universe-volatility.function.ts`
  and the symbol-grouping logic in `volatility-query.function.ts` (lines around 78) are deleted
  once the distribution-history path is wired and the unit + e2e tests pass.

- **Window selection.** `dividendhistory.net` returns dated per-share distribution rows. Apply
  the same windowing the legacy code applied: filter rows where `date >= now - 1y` for
  short-term, and `date >= now - 5y` for long-term, then map to amount arrays for
  `calculateVolatility()`.

- **Trigger preservation.** The trigger wiring added in Epic 85 (Story 85.2 —
  `wire-volatility-recalculation-triggers`) stays in place. Those triggers continue to call
  `recalculateUniverseVolatility(universeId)`; only that function's *internal* data source
  changes. No new trigger sites are required because dividend-history fetches already happen
  inside the universe-sync flow.

- **Insufficient-history behaviour.** The existing `recalculate-universe-volatility.function.spec.ts`
  case `'writes insufficient-history when no distribution history exists'` (line ~96) must
  continue to pass — the new code path must produce the same `null`/`unknown` category when the
  distribution payload is empty or below the minimum row count.

## Epic 88: Volatility From Distribution History (All Universe Symbols)

**Goal:** Replace the broker-deposit-driven volatility calculation with one that consumes the
per-share dividend distribution history already fetched from `dividendhistory.net` during
universe sync. The result is short-term and long-term volatility persisted to the `universe`
table (columns added in Epic 85) for **every** symbol the universe knows about — held or not —
so the Vol and SVol columns become meaningful for buy/sell decisions across the entire screener.

### Story 88.3: Wire Volatility Recalculation Into the Universe Distribution-History Fetch

**Slug:** `88-3-wire-distribution-history-volatility-into-universe-retrieval`

As Dave,
I want volatility to be (re)calculated and persisted **for every symbol** at the moment the
universe-sync flow fetches its distribution history from dividendhistory.net,
so that after a universe sync the `universe` table has correct, current `volatility_long` and
`volatility_short` values for every symbol — not just the ones I have positions in.

**Acceptance Criteria:**

1. **Given** the call site identified in Story 88.1,
   **When** the universe-sync flow successfully fetches distribution history for a symbol via
   `fetchDividendHistory(symbol)`,
   **Then** the returned payload is passed straight into the refactored
   `recalculateUniverseVolatility(universeId, history)` (no re-fetch) before the per-symbol
   sync step completes.

2. **Given** the universe contains symbols the user has never held a position in,
   **When** universe sync runs end-to-end,
   **Then** every such symbol's row in `universe` has non-null `volatility_long` and
   `volatility_short` values (or the `null`/`unknown` category if dividendhistory.net returned
   no rows for it) and a fresh `volatility_calculated_at` timestamp.

3. **Given** `fetchDividendHistory(symbol)` fails (404, network error, or empty body),
   **When** universe sync continues,
   **Then** the symbol's volatility columns are written as the `null`/`unknown` category and
   the failure is logged using the existing universe-sync error path — sync does not abort.

4. **Given** the existing trigger wiring from Epic 85 Story 85.2,
   **When** any other caller invokes `recalculateUniverseVolatility(universeId, ...)` (for
   example after a manual symbol add),
   **Then** that caller is updated to also fetch distribution history first (or to pass an

5. **Given** the wiring is complete,
   **When** `pnpm all` runs,
   **Then** all unit tests pass, including any sync-flow specs that needed updating for the
   new per-symbol recalculation step.

**Definition of Done:**

- [ ] No second call to `fetchDividendHistory()` introduced anywhere
