# dms-workspace — Epic Breakdown (2026-05-13)

## Overview

Three new epics derived from the 2026-05-13 epic descriptions file. This batch extends the
project backlog from **Epic 103** through **Epic 105**. All three are bug fixes against
already-shipped surfaces:

- **Epic 103 — Add New Symbol on Universe Screen (Validation Polarity Fix):** The "Add"
  button on the Universe screen's *Add Symbol* modal never enables, even after the form is
  filled in correctly. The most likely cause is that the modal is using the same symbol
  validator as the *Open Positions Add* modal, where the rule is "the symbol must already be
  in the Universe list" — the exact opposite of what the Universe Add modal needs ("the symbol
  must NOT already be in the Universe list"). The fix must correct the Universe Add modal
  without regressing the Open Positions Add modal, which is currently behaving correctly.

- **Epic 104 — Closing a Position Removes It from Open Positions Immediately:** On the Open
  Positions screen, when Dave closes a position by setting `Mst Rcnt Sll Dt` to a date and
  `Mst Rcnt Sell $` to a positive value, the row should disappear from Open Positions as soon
  as the change is saved. Today the row only disappears after a manual page refresh — the
  client-side store / list filter is not reacting to the saved change. This epic fixes the
  immediate-removal behaviour and pins it with a regression test.

- **Epic 105 — Janky Scrolling After Account / Filter Change (Round 8):** The scrolling work
  done in Epic 101 (and predecessors 29, 31, 44, 60, 64, 87) successfully eliminated the
  artifacts on freshly loaded screens. They have come back, but only after Dave changes the
  active account (or, by extension, applies/clears a filter) on the Universe screen — the
  artifacts then return on that same screen until the next full page load. This is a new
  failure mode: something about the account-change / filter-change pipeline resets or fails
  to reset state that the scrolling fix from Epic 101 depends on. This epic isolates that
  state, fixes it, and adds regression coverage targeted at the context-change scenario
  specifically (the previous regression suite only covered the freshly-loaded case).

All stories follow existing project conventions:

- **Angular 21 zoneless** — `inject()`, `OnPush`, signal-first
- **SmartNgRX / SmartSignals** for state management
- **Fastify** backend, **Prisma** ORM (SQLite dev / PostgreSQL prod)
- **Vitest** for unit tests, **Playwright** for E2E (Chromium + Firefox)
- **Tests are authoritative** — do not change tests unless a test is asserting something we
  are explicitly changing
- **`pnpm all`** must pass after every story
- **Playwright MCP server** must be used to reproduce and verify all UI failures

### Additional Requirements (Architecture / Context)

- **Shared Add-symbol validator (Epic 103):** Per the issue reporter, the Universe Add modal
  and the Open Positions Add modal share validation logic. The two modals require **opposite**
  polarity on the "is symbol in Universe?" check: Universe Add wants `NOT IN`, Open Positions
  Add wants `IN`. The fix must parameterise the shared validator (or split into two
  intention-named validators) — not duplicate-and-diverge or special-case by route. Likely
  location: an Angular validator factory under `apps/dms-material/src/app/.../validators/`
  consumed by both add-modal forms. Investigation must confirm the actual shared file.

- **Open Positions list reactivity (Epic 104):** Open Positions rows are sourced from the
  server-joined Trade DTO (see Epic 95/97). The "open" vs "sold" filter is applied either
  server-side (the Open Positions endpoint excludes closed trades) or client-side (the
  SmartNgRX/SmartSignals selector filters by sell-date / sell-price). The bug indicates that
  after a save, the client's view of the row does not re-evaluate against the open/closed
  predicate. Investigation must identify whether (a) the save mutation does not update the
  store entity, (b) the store entity updates but the Open Positions selector is not reacting
  to the changed fields, or (c) the open/closed split is server-side and the screen is not
  refetching after the save. The fix must target the actual broken layer.

- **Scrolling Round 8 search space (Epic 105):** Epic 101 fixed scrolling for freshly loaded
  screens. The regression returns specifically when the active account or a filter changes
  *without a full reload*. Likely candidates: the CDK `cdk-virtual-scroll-viewport`'s
  internal data-source / range-renderer is not being reset when the underlying signal swaps
  out, leaving stale row-height measurements; the sticky header's containing block is being
  re-created (or not re-created) by a structural directive on account change, breaking the
  Epic-101 fix; the SmartNgRX/SmartSignals selector returns a new array reference but the
  same row identities, causing the virtual scroller to keep stale viewport offsets; or an
  ancestor `transform`/`will-change`/`contain` is being applied conditionally during the
  account-change loading state. Investigation must enumerate which of these is in play before
  coding a fix.

- **Prior scrolling epics for context (Epic 105):** Epics 29, 31, 44, 60, 64, 87, and 101 all
  addressed scrolling. Their epics files live in `_bmad-output/planning-artifacts/` (search
  for `epics-*.md` containing those numbers). Story 105.1 must summarise what each prior
  attempt covered so we don't waste effort retrying an already-tried hypothesis.

- **Playwright MCP for UI reproduction (NFR3):** All three epics include reproduction steps
  that must be driven via the Playwright MCP server (toggling accounts, filling modals,
  saving rows, slow-scrolling) so that the failure can be observed and the fix verified
  independently of unit tests.

## Epic 103: Add New Symbol on Universe Screen (Validation Polarity Fix)

after Dave fills in a valid new symbol. The likely cause is that the modal is using the same

Add, and adds regression coverage for both modals.

### Story 105.3: Regression Test Suite — Scrolling Stays Clean After Account / Filter Change

**Slug:** `105-3-regression-suite-scrolling-after-account-filter-change`

As a developer,
I want a focused E2E regression suite that, on every virtual-scrolled screen, performs a
context change (account swap and/or filter apply-clear) and then drives slow scrolling and
asserts the same header invariants as Epic 101's suite,
So that any future change that breaks the Round-8 fix fails CI immediately and Round 9 of
this epic never starts.

**Acceptance Criteria:**

**Given** every screen identified in the Story 105.1 matrix,
**When** the regression test runs against that screen,
**Then** the test (a) loads the screen, (b) changes the active account (or applies/clears a
filter, depending on the screen), (c) scrolls slowly through the resulting virtual list, and
(d) asserts the header element remains at the top of its scroll container and does not
visually overlap the parent header at any frame. (Use Playwright's bounding-box and
screenshot assertions where appropriate.)

**Given** Chromium and Firefox,
**When** the suite runs in both,
**Then** all assertions pass in both browsers.

**Given** the suite is committed,
**When** the CI pipeline runs,
**Then** the suite is part of the normal `pnpm all` execution and is not skipped.

**Given** a deliberate regression is hand-tested locally (e.g. revert the Story 105.2 fix),
**When** the suite runs,
**Then** at least one assertion fails — confirming the suite would actually catch a
regression for the context-change scenario specifically.

**Definition of Done:**

- [ ] Regression test exists for every screen in the Story 105.1 matrix
- [ ] Tests trigger an account change and/or filter apply-clear before the slow scroll
- [ ] Tests assert header-position invariants during slow scroll
- [ ] Tests pass on both Chromium and Firefox
- [ ] Tests are not skipped; included in `pnpm all`
- [ ] Manually verified to fail when the Story 105.2 fix is reverted
- [ ] `pnpm all` passes
