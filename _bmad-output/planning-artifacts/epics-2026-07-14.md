# dms-workspace - Epic Breakdown for Table Layout Improvements

## Overview

This document provides the complete epic and story breakdown for dms-workspace, decomposing the requirements from the epic description file into implementable stories.

**Critical constraint:** Unit test stories appear before implementation stories. Implementation stories will un-skip the skipped unit tests. All stories for unit tests, implementation, and e2e tests are included for every change.

## Requirements Inventory

### Functional Requirements

From the epic description (no formal FR/FNR list exists — this is a single-improvement epic):

- Replace `<table mat-table>` syntax with `<mat-table>` (no `<table>` wrapper) using `<ng-container matColumnDef>` column definitions
- Revert `base-table` component (apps/dms-material/src/app/shared/components/base-table) from DIVs back toward the original table-based implementation, then convert to `<mat-table>` format
- Review and update all consumers/inheritors of `base-table` to work with the new column-definition approach
- Remove the header/table separation that was previously needed
- Write/verify unit tests for base-table BEFORE any code changes (skip tests at story creation; un-skip during implementation)
- Write/verify unit tests for all consumer components BEFORE any code changes (skip tests at story creation; un-skip during implementation)
- Update E2E regression tests to work with the new `<mat-table>` DOM structure
- Update E2E consumer tests to work with the new `<mat-table>` DOM structure

## Epic List

### Epic 1: Migrate base-table to `<mat-table>` Column Definition Pattern

**Goal:** Replace the current `<table mat-table>` / DIV-based `base-table` component with the `ng-container matColumnDef` pattern, eliminating the need for header/table separation and improving layout stability.

**Scope:** The `base-table` component, all consumer components, all unit tests for base-table and consumers, and all E2E tests that interact with base-table or its consumers.

**Order constraint:** Story 1.1 (unit test scaffolds) → Story 1.3 (implementation). Stories 1.2, 1.4, 1.5, 1.6 follow the same pattern.

### Story 1.1: Unit test scaffolds for base-table component (skip)

As a developer,
I want to create/verify unit test scaffolds and assertions for the base-table component BEFORE any code changes,
So that tests exist before implementation and the dev agent can verify correctness after changes.

**Acceptance Criteria:**

**Given** the current DIV-based `base-table.component.spec.ts` exists
**When** I review and enrich it with assertions covering all public API behaviors (9 inputs, 4 outputs, sorting, selection, filterRowTemplate, cellTemplate, virtual scroll)
**Then** the test file compiles and all assertions are written with `.skip` so they are not executed yet
**And** the scaffold covers every testable method, ViewChild, ContentChild, and computed property
**And** the test file is ready for the dev agent to un-skip once implementation is complete

**Story metadata:** skip. Unit test scaffolds with `.skip` on all assertions. Will be un-skipped during Story 1.3.

### Story 1.2: Unit test scaffolds for base-table consumer components (skip)

As a developer,
I want to create/verify unit test scaffolds and assertions for all consumer components BEFORE any code changes,
So that tests exist before implementation and can verify consumers continue to work after changes.

**Acceptance Criteria:**

**Given** the consumer component spec files exist (dividend-deposits, open-positions, sold-positions, global-universe)
**When** I review and enrich each spec with assertions covering binding to base-table inputs/outputs, cellTemplate rendering, filterRowTemplate rendering, and template stability
**Then** each test file is written with `.skip` on the new assertions
**And** the scaffolds cover each consumer's unique behavior (editable cells, delete action, symbol search filter, viewChild(BaseTableComponent) reference)
**And** the test files are ready for the dev agent to un-skip once implementation is complete

**Story metadata:** skip. Unit test scaffolds with `.skip` on all assertions. Will be un-skipped during Story 1.4.

### Story 1.3: Convert base-table component to `<mat-table>` with `ng-container matColumnDef`

As a developer,
I want to rewrite the `base-table` component template and related code to use `<mat-table [dataSource]="dataSource">` with `<ng-container matColumnDef>` columns,
So that the table layout follows Angular Material's recommended pattern and avoids sticky-header complications.

**Acceptance Criteria:**

**Given** the original table-element structure from commit 36ee7e7cb9910d9ac54cb2c719ffd36cef649520
**When** the component template is converted to `<mat-table>` (no `<table>` wrapper) with `<ng-container matColumnDef>` for each column
**Then** the component uses `<mat-header-cell *matHeaderCellDef>`, `<mat-cell *matCellDef="let row">`, `<mat-header-row *matHeaderRowDef>`, and `<mat-row *matRowDef="let row; columns">` directly inside `<mat-table>`
**Then** the previously skipped unit tests from Story 1.1 are now un-skipped (`.skip` removed)
**And** the component still exposes the same public API (9 inputs, 4 outputs, ContentChildren, ViewChildRefs, 13 methods)
**And** CDK virtual scrolling still works
**And** horizontal scroll sync still works
**And** selection (single/multi) still works
**And** the component compiles without errors

**Story metadata:** Un-skips `.skip` in Story 1.1 unit tests. This is the main implementation story.

### Story 1.4: Update base-table consumer components for the new DOM structure

As a developer,
I want to update all consumer components that use `base-table`,
So that they work with the new `<mat-table>` column-definition pattern without breaking existing behavior.

**Acceptance Criteria:**

**Given** the updated `base-table` from Story 1.3
**When** all 5 consumers are reviewed
**Then** consumer templates still work identically (no template changes needed if public API is preserved)
**And** consumer SCSS still targets `dms-base-table { display: block; height: 100%; width: 100%; }` correctly
**And** the previously skipped unit tests from Story 1.2 are now un-skipped (`.skip` removed)
**And** all un-skipped consumer tests pass
**And** no TypeScript compilation errors exist

**Story metadata:** Un-skips `.skip` in Story 1.2 unit tests. This is the consumer implementation story.

### Story 1.5: Update base-table E2E regression tests for new `<mat-table>` DOM

As a developer,
I want to update the base-table E2E regression tests to work with the new `<mat-table>` DOM structure,
So that automated table layout testing continues to catch regressions with the new implementation.

**Acceptance Criteria:**

**Given** the new `<mat-table>` DOM structure from Story 1.3
**When** `base-table-two-region-regression.spec.ts` is updated
**Then** all selectors are changed from DIV-based classes (`.dms-table-shell`, `.dms-table-header`, etc.) to mat-table equivalents (`mat-header-row`, `mat-row`, `mat-header-cell`, `mat-column-`, etc.)
**And** the test still verifies all previous behaviors (vertical header scroll, body scroll, header-body scroll sync, virtual scroll)
**And** the test passes in chromium and firefox

**Story metadata:** e2e. Update base-table E2E regression test file for mat-table DOM.

### Story 1.6: Update consumer E2E tests for new `<mat-table>` DOM

As a developer,
I want to update E2E tests for all affected consumer components to work with the new `<mat-table>` DOM structure and selectors,
So that consumer testing continues to verify correct table rendering after the migration.

**Acceptance Criteria:**

**Given** the new `<mat-table>` DOM structure from Story 1.3
**When** `dividend-precision.spec.ts` is updated
**Then** `dms-base-table` visibility assertions still work (they target the host component, not internal DOM)
**When** `account-sort-filter-persistence*.spec.ts` files are updated
**Then** all locators are updated from DIV-based selectors to mat-table equivalents
**And** all consumer E2E tests pass in chromium and firefox
**And** all sorting, filtering, selection, and cell template behaviors are still tested and verified

**Story metadata:** e2e. Update all consumer E2E test files.
