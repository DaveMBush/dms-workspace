# Story 1.6: Update consumer E2E tests for new `<mat-table>` DOM

Status: Approved

## Story

As a developer,
I want to update E2E tests for all affected consumer components to work with the new `<mat-table>` DOM structure and selectors,
So that consumer testing continues to verify correct table rendering, sorting, filtering, and cell template behaviors after the migration.

## Acceptance Criteria

1. [The E2E test file `dividend-precision.spec.ts` has all necessary selectors updated for mat-table DOM]
2. [The E2E test files `account-sort-filter-persistence*.spec.ts` have all necessary selectors updated for mat-table DOM]
3. [Consumer E2E tests pass in chromium AND firefox]
4. [All sorting, filtering, selection, and cell template behaviors are still tested and verified]
5. [No regression in existing test behavior or assertions]

## Dev Notes

### E2E Test Files to Update

1. `apps/dms-material-e2e/src/dividend-precision.spec.ts`
   - Tests dividend precision/currency formatting in the base-table
   - May have `dms-base-table` or internal DOM selectors

2. `apps/dms-material-e2e/src/account-sort-filter-persistence*.spec.ts`
   - Tests sort/filter state persistence across navigations
   - Likely uses selectors targeting inside consumer components + base-table
   - These selectors may have `.dms-*` classes that need mat-table equivalents

### Selector Migration Reference (same as Story 1.5)

| Old Selector | New Selector |
|---|---|
| `.dms-header-cell` | `mat-header-cell` or `mat-column-{field}` |
| `.dms-body-cell` | `mat-cell` |
| `.dms-body-row` | `mat-row` |
| `.dms-checkbox` | `mat-checkbox` |
| `input[type="search"]` (in filterRowTemplate) | remains the same (it's an HTML element, not a DIV class) |
| `[data-testid="base-table-header"]` | verify target still valid |

### Anti-Patterns to Avoid

- **DON'T** change test logic/assertions — only the selectors
- **DON'T** modify the base-table E2E test (that's Story 1.5's job)
- **DON'T** modify consumer component code — only E2E test files
- **DON'T** remove any test assertions
- **DON'T** assume test files in `e2e/` are the only ones affected — also check component-level specs for any E2E integration tests
